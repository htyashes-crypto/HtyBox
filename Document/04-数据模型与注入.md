# 04 · 数据模型、目录发现与注入

本文定义 HtyBox 如何**发现**磁盘上的 skill/memory、如何**建模**，以及拖拽落下时按终端类型**注入**什么文本。这是"喂给终端"这一核心交互的规则中心。

---

## 1. Skill 模型与发现

### 1.1 文件形态

每个 skill 是一个目录下的 `SKILL.md`，YAML frontmatter 提供 `name` 与 `description`：

```markdown
---
name: frontend-design
description: Create distinctive, production-grade frontend interfaces...
---
（正文：技能说明）
```

### 1.2 发现路径（三类来源）

| 来源 | 扫描位置（glob） | 调用串 `invoke` |
|---|---|---|
| **User** | `~/.claude/skills/*/SKILL.md` | `/<name>` |
| **Project** | `<当前项目>/.claude/skills/*/SKILL.md` | `/<name>` |
| **Plugin** | `~/.claude/plugins/marketplaces/**/skills/*/SKILL.md` | `/<plugin>:<name>` |

> 说明：本机当前 `~/.claude/skills/` 为空，skill 主要来自插件市场（如 `frontend-design`、`mcp-server-dev` 等）。Plugin 调用要用 `插件名:技能名` 的命名空间形式（如 `/frontend-design:frontend-design`），所以模型里要带 `source`，由它推导 `invoke`。
>
> 插件名的提取：从插件目录路径里解析（`plugins/marketplaces/<market>/plugins/<plugin>/skills/<skill>/SKILL.md` 或 `external_plugins/<plugin>/skills/<skill>/SKILL.md`）。具体 `invoke` 串需在 **M5** 用真实 Claude Code 校验后定稿（不同安装方式的命名空间可能不同），因此设计为**可配置模板**。

### 1.3 数据结构

```ts
type SkillSource =
  | { kind: "user" }
  | { kind: "project" }
  | { kind: "plugin"; plugin: string };

interface Skill {
  name: string;
  description: string;
  path: string;        // SKILL.md 绝对路径
  source: SkillSource;
  invoke: string;      // 推导出的调用串，如 "/frontend-design"
}
```

## 2. Memory 模型与发现

### 2.1 文件形态

```markdown
---
name: user-role
description: User is taking over the Arena project from a colleague who left
metadata:
  type: user            # user | feedback | project | reference
---
（正文）
```

同目录有一个 `MEMORY.md` 索引（每行一条指向某个 memory 文件），它不是 memory 条目本身。

### 2.2 发现路径

```
~/.claude/projects/<project-slug>/memory/
   ├─ MEMORY.md          # 索引（单独处理或忽略）
   ├─ user_role.md       # 一条 memory
   ├─ project_xxx.md
   └─ ...
```

扫描 `*.md`（排除 `MEMORY.md`），逐个解析 frontmatter。

### 2.3 项目 slug 算法（关键）

memory 是**按项目隔离**的。项目根路径要转成目录名 `<slug>`。从本机现存目录反推出的规则：

> **把绝对路径中的 `:`、`\`、`/`、`_` 全部替换成 `-`，大小写保持不变。**

验证（用本机实际存在的目录核对）：

| 项目根路径 | 计算出的 slug | 实际目录 | ✓ |
|---|---|---|---|
| `G:\hty_workflows` | `G--hty-workflows` | `G--hty-workflows` | ✓ |
| `C:\Users\admin` | `C--Users-admin` | `C--Users-admin` | ✓ |
| `d:\igpworks\arena` | `d--igpworks-arena` | `d--igpworks-arena` | ✓ |
| `e:\HtyEngine` | `e--HtyEngine` | `e--HtyEngine` | ✓ |
| `g:\Learning-Self` | `g--Learning-Self` | `g--Learning-Self` | ✓ |

`G:\` 变成 `G--` 的原因：`:` → `-` 再 `\` → `-`，两个分隔符连续替换得到 `--`。

```rust
fn project_slug(abs_path: &str) -> String {
    abs_path.chars()
        .map(|c| match c { ':' | '\\' | '/' | '_' => '-', other => other })
        .collect()
}
```

> ⚠️ 该算法是从现状反推的约定，Claude 未来可能调整。**兜底设计**：配置里允许手动指定/覆盖 memory 根目录；同时提供 `list_projects`（直接列 `~/.claude/projects/*` 真实目录名）让用户从下拉框选，而非只依赖反推。

### 2.4 数据结构

```ts
interface MemoryItem {
  name: string;
  description: string;
  memType: "user" | "feedback" | "project" | "reference" | string;
  path: string;          // 绝对路径
  projectSlug: string;
}
```

### 2.5 作用域问题（设计决策）

多个终端可能 `cd` 在不同项目里，"该显示哪个项目的 memory？"

- **v1**：右栏顶部一个 `ProjectSelector`（全局"当前项目"），默认 = 应用启动目录（`G:\hty_workflows`）。memory 列表跟随它。
- **v2（增强）**：跟随**当前聚焦终端的工作目录**自动切换 memory 作用域 —— 聚焦哪个终端，右栏就显示那个项目的 memory。需要后端上报每个终端的 cwd（可在 spawn 时记录，或解析 shell 标题）。

## 3. 终端 Profile

Profile 决定一个终端启动什么、cwd 是什么、以及**注入时用哪套模板**。

```ts
interface TerminalProfile {
  id: string;                  // "claude" | "codex" | "pwsh" | "bash" | 自定义
  label: string;              // 显示名
  shell: string;              // 实际启动的 shell, 如 "powershell.exe"
  args: string[];
  cwd?: string;               // 不填则用全局当前项目
  env?: Record<string, string>;
  launchCmd?: string;         // 启动后自动发送, 如 "claude\r" / "codex\r"
  agentKind: "claude" | "codex" | "shell";  // 决定注入模板
}
```

内置 Profile（可在设置里增改）：

| Profile | shell | launchCmd | agentKind |
|---|---|---|---|
| Claude Code | `powershell.exe` | `claude\r` | `claude` |
| Codex | `powershell.exe` | `codex\r` | `codex` |
| PowerShell | `powershell.exe` | — | `shell` |
| Bash | `bash` (Git Bash) | — | `shell` |

> 采用"先起 shell 再自动发命令"的方式启动 claude/codex，规避 Windows 上 `.cmd` 直接当 exe 启动找不到的问题（见 [02 §3.3](./02-后端Rust设计.md)）。

## 4. 注入模板矩阵（核心）

拖拽落下时，后端 `inject::build_text(agentKind, item)` 按下表生成文本：

| 被拖项 | `agentKind=claude` | `agentKind=codex` | `agentKind=shell` |
|---|---|---|---|
| **Skill** | `/<invoke>`（如 `/frontend-design`） | `@<SKILL.md 绝对路径>` | `<SKILL.md 绝对路径>`（纯文本，用户自行处理） |
| **Memory** | `@<memory 绝对路径>` | `@<memory 绝对路径>` | `<memory 绝对路径>` |

设计依据：
- **Claude Code** 原生支持 `/skill-name` 斜杠命令调用技能，支持 `@path` 引用文件 → 用引用最省 token，且文件内容更新后引用自动指向最新。
- **Codex** 没有斜杠技能体系，但支持以文件路径作为上下文 → 统一用 `@路径`。
- **shell** 无 AI，注入纯路径，留给用户决定（例如手动 `cat`）。

> ⚠️ `@路径` 与 `/插件:技能` 的**确切语法**依赖 claude/codex 当前版本，必须在 **M4/M5** 用真实 CLI 校验。所以模板做成**配置可改**（见 §5 的 `injectTemplates`），代码里只放默认值，校验后调默认即可，不必改代码。

```rust
// inject.rs（示意）
pub fn build_text(kind: &AgentKind, item: &InjectPayload, tpl: &InjectTemplates) -> String {
    match (kind, &item.kind) {
        (AgentKind::Claude, ItemKind::Skill) => tpl.claude_skill.replace("{invoke}", &item.invoke),
        (AgentKind::Claude, ItemKind::Memory) => tpl.claude_memory.replace("{path}", &item.path),
        (AgentKind::Codex,  _)               => tpl.codex_any.replace("{path}", &item.path),
        (AgentKind::Shell,  _)               => item.path.clone(),
    }
}
```

注入流程：
1. 前端 drop → `invoke("inject_item", { termId, item, submit })`
2. 后端查该终端 Profile 的 `agentKind` → `build_text` → 得到文本
3. `PtyManager.write(termId, text)`；若 `submit==true` 再 `write(termId, "\r")`
4. 用户在终端里看到注入的引用，确认/补充后回车（或拖拽时已按 Shift 自动发送）

## 5. 配置文件

通过 `tauri-plugin-store` 持久化为 JSON（如 `htybox.config.json`）：

```jsonc
{
  "layout": {
    "columnSizes": [240, 1200, 240],     // 三栏宽度
    "dock": { /* dockview toJSON() 的布局快照 */ }
  },
  "currentProject": "G:\\hty_workflows",  // memory 作用域
  "scanRoots": {
    "userSkills": "~/.claude/skills",
    "pluginSkills": "~/.claude/plugins/marketplaces",
    "memoryBase": "~/.claude/projects"     // 兜底可改
  },
  "profiles": [ /* TerminalProfile[] */ ],
  "injectTemplates": {
    "claudeSkill": "/{invoke}",
    "claudeMemory": "@{path}",
    "codexAny": "@{path}",
    "autoSubmit": false                     // 默认不自动回车；Shift 落下可临时覆盖
  }
}
```

- 启动时 `get_config` 读入，缺省字段用内置默认值填充并回写。
- 设置面板（v1 可简化为编辑该 JSON / 基础表单）允许改注入模板，应对 CLI 语法变化时**无需改代码**。
