# 07 · 工作区(Workspace)管理

> 对应原型顶部的 `workspace1 / workspace2` 标签。本文定调工作区概念，并明确一个关键生命周期决策。

---

## 1. 概念:工作区 = 完整环境 + Agent 团队边界

一个 **Workspace** 是一套**自包含的工作环境**，切换顶部 tab 就整体换一套：

- 自己的**终端集合**（各终端的 Profile + cwd + 是否 Agent 角色）
- 自己的 **dockview 分屏/标签布局**
- 自己的 **memory 作用域**（当前项目）
- 自己的 **skill 过滤/置顶**
- （关键）自己的**多 Agent 协作团队** —— 见 [08](./08-多Agent协作设计.md)

> **统一模型**：一个 workspace 就是"**一个 Agent 团队 + 它们的共享上下文与协作范围**"。多 Agent 编排只在同一 workspace 内发生。这样"工作区"和"团队"是同一个边界，概念不重复。

典型用法：`workspace1` = 给项目 A 的一组协作 agent；`workspace2` = 给项目 B 的另一组。两组互不干扰、各自后台运行。

---

## 2. 关键决策:切换时 PTY 后台常驻 ⚠️

**决策**：切到别的 workspace 时，被切走的 workspace 里的**终端进程(PTY)继续在后台运行**，仅隐藏 UI，不杀进程。

**理由**：多 Agent 在后台跑着活，一切 tab 就被杀掉则完全没法用。Agent 协作往往耗时，必须能"切走看别的、回来还在跑"。

**代价与配套**：
- 后台进程占用资源 → 每个 workspace tab 上显示**运行指示**（运行中 agent 数 / 是否有活动），并提供**手动挂起**（暂停/冻结该 workspace 的全部 agent）与**关闭**（杀掉全部 PTY）。
- xterm 实例可在隐藏时**分离 DOM 但保留 PTY 与 scrollback**，切回时重新挂载并补绘（xterm 支持 serialize/restore，或保留实例只隐藏容器）。
- 应用退出时遍历所有 workspace 的 PTY 统一 kill，防僵尸进程（[06 R7](./06-风险测试与打包.md)）。

---

## 3. 数据模型

```ts
interface Workspace {
  id: string;
  name: string;                  // tab 显示名，可重命名
  project: string;               // memory 作用域 / 多数终端默认 cwd
  dockLayout: object;            // dockview toJSON() 快照
  terminals: TerminalSpec[];     // 该工作区的终端定义
  team?: TeamConfig;             // 多 Agent 团队配置（见 08），无则纯多终端
  createdAt: string;
}

interface TerminalSpec {
  termId: string;
  profileId: string;             // claude / codex / pwsh / bash
  cwd: string;
  role?: "lead" | "worker" | null;   // 参与多 Agent 编排时的角色
  agentId?: string;              // 在团队里的稳定 id（见 08 身份）
}
```

配置持久化（接 [04 §5](./04-数据模型与注入.md) 的 `tauri-plugin-store`）：

```jsonc
{
  "workspaces": [ /* Workspace[] */ ],
  "activeWorkspaceId": "ws-1",
  // ...其余全局配置
}
```

> 进程态（PTY、xterm 实例、agent 运行状态）是**运行时**的，不入配置；配置只存"结构"。重启后恢复布局与终端定义，但 agent 进程是重新拉起（会话不自动续，列入 [06 v2](./06-风险测试与打包.md) 的"会话恢复"）。

## 4. UI 与交互

- 顶部一行 **WorkspaceTabs**：`+ 新建`、双击重命名、右键菜单（复制、挂起、关闭）。
- 每个 tab 显示名称 + **运行指示徽标**（如 `▶3` 表示 3 个 agent 在跑；`⏸` 表示已挂起）。
- 切换 tab：保存当前 dockLayout → 隐藏当前栏内容 → 挂载目标 workspace 的布局（PTY 不动）。
- 新建 workspace：选模板（空白 / 复制当前 / **从团队库选一个 Team 一键开启** —— 把 Team 实例化为本 workspace 的协作团队，见 [08 §2](./08-多Agent协作设计.md)）。

布局位置（在 [01 §5](./01-产品与架构.md) 的三栏之上加一行）：

```
┌──────────────────────────────────────────────────────────┐
│ [workspace1 ▶3] [workspace2 ⏸] [ + ]        ← WorkspaceTabs│
├──────────┬────────────────────────────────┬──────────────┤
│  Skills  │     终端区 (dockview)           │   Memory     │
│  ...     │     ...                         │   ...        │
└──────────┴────────────────────────────────┴──────────────┘
```

---

## 5. 与 skill / memory 作用域的关系

- **memory**：跟随 workspace 的 `project` 字段决定作用域（替代 [04 §2.5](./04-数据模型与注入.md) 里"全局当前项目"——作用域提升到 workspace 级）。
- **skill**：user/plugin skill 是全局的；project skill 跟随 workspace 的 project。

---

## 6. 里程碑（插入主路线）

定位为 **M2.5**（在多终端 M2 之后、目录 M3 并行期即可做，因为它主要是状态分层与持久化）：

- [ ] Workspace 数据模型 + 顶部 `WorkspaceTabs`（新建/切换/重命名/关闭）
- [ ] 切换时保存/恢复 dockLayout，**PTY 后台常驻**（隐藏 UI 不杀进程）
- [ ] 运行指示徽标 + 手动挂起/关闭
- [ ] memory/skill 作用域提升到 workspace 级
- [ ] 多 workspace 持久化与启动恢复（结构层）

**验收**：开两个 workspace，各跑各的终端；来回切 tab，被切走的进程仍在跑、scrollback 不丢；重启后结构恢复。

> 多 Agent 团队配置(`team`)的设计见 [08](./08-多Agent协作设计.md)；本文只负责"工作区作为团队容器"这一层。
